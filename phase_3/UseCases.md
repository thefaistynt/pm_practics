# Описание проекта
По итогам Фазы 1 был выбран проект «Система бронирования аудиторий, переговорных и техники».

# Use Cases

## 1. Регистрация нового пользователя
### Предусловия:
Пользователь впервые заходит в систему.

### Триггер:
Пользователь нажимает кнопку «Зарегистрироваться».

### Основной сценарий:
1. Система предлагает ввести email (или корпоративный логин) и пароль.
2. Пользователь вводит данные.
3. Система проверяет, что пользователь с таким email не зарегистрирован.
4. Система отправляет код подтверждения на email.
5. Пользователь вводит код подтверждения.
6. Система предлагает выбрать роль (сотрудник / администратор).
7. Пользователь подтверждает регистрацию.
8. Система перенаправляет пользователя на главный экран.

### Результат:
Создан новый аккаунт пользователя.

### Альтернативный сценарий:
1. Email уже зарегистрирован: система предлагает выполнить вход.
2. Неверный код подтверждения: система выводит сообщение об ошибке.

## 2. Авторизация пользователя
### Предусловия:
Пользователь зарегистрирован в системе.

### Триггер:
Пользователь нажимает кнопку «Войти».

### Основной сценарий:
1. Система запрашивает email и пароль.
2. Пользователь вводит учетные данные.
3. Система проверяет корректность данных.
4. Система авторизует пользователя и открывает главный экран.

### Результат:
Пользователь вошёл в систему.

### Альтернативный сценарий:
Неверный логин или пароль: система выводит сообщение об ошибке.

## 3. Просмотр списка доступных ресурсов
### Предусловия:
Пользователь авторизован.

### Триггер:
Пользователь открывает раздел «Ресурсы».

### Основной сценарий:
1. Система отображает список ресурсов (аудитории, переговорные, техника).
2. Для каждого ресурса показывается статус доступности.
3. Пользователь применяет фильтры (тип ресурса, вместимость, дата).

### Результат:
Пользователь видит актуальный список доступных ресурсов.

### Альтернативный сценарий:
Нет доступных ресурсов: система отображает соответствующее сообщение.

## 4. Поиск ресурса по параметрам
### Предусловия:
Пользователь авторизован.

### Триггер:
Пользователь вводит параметры поиска (дата, время, тип ресурса).

### Основной сценарий:
1. Система принимает параметры поиска.
2. Система анализирует расписание бронирований.
3. Система выводит список доступных ресурсов.

### Результат:
Пользователь получает список ресурсов, подходящих под заданные условия.

### Альтернативный сценарий:
Подходящих ресурсов нет: система предлагает изменить параметры поиска.

## 5. Бронирование аудитории или переговорной
### Предусловия:
1. Пользователь авторизован.
2. Ресурс доступен на выбранное время.

### Триггер:
Пользователь нажимает кнопку «Забронировать».

### Основной сценарий:
1. Система открывает форму бронирования.
2. Пользователь указывает дату, время начала и окончания.
3. Пользователь подтверждает бронирование.
4. Система проверяет отсутствие конфликтов.
5. Система сохраняет бронирование.

### Результат:
Ресурс успешно забронирован.

### Альтернативный сценарий:
В выбранное время ресурс уже занят: система предлагает выбрать другое время.

## 6. Бронирование техники
### Предусловия:
1. Пользователь авторизован.
2. Техника доступна.

### Триггер:
Пользователь выбирает технику и нажимает «Забронировать».

### Основной сценарий:
1. Система показывает форму с датой и временем использования.
2. Пользователь указывает период бронирования.
3. Система проверяет доступность техники.
4. Система подтверждает бронирование.

### Результат:
Техника забронирована на выбранный период.

### Альтернативный сценарий:
Техника недоступна: система предлагает альтернативные варианты.

## 7. Просмотр своих бронирований
### Предусловия:
Пользователь авторизован.

### Триггер:
Пользователь открывает раздел «Мои бронирования».

### Основной сценарий:
1. Система отображает список всех активных и прошедших бронирований.
2. Пользователь может открыть детали бронирования.

### Результат:
Пользователь видит все свои бронирования.

### Альтернативный сценарий:
У пользователя нет бронирований: система показывает пустой список.

## 8. Отмена бронирования
### Предусловия:
1. Пользователь авторизован.
2. Бронирование активно.

### Триггер:
Пользователь нажимает кнопку «Отменить бронирование».

### Основной сценарий:
1. Система запрашивает подтверждение отмены.
2. Пользователь подтверждает действие.
3. Система удаляет бронирование.
4. Ресурс становится доступным для других пользователей.

### Результат:
Бронирование отменено.

### Альтернативный сценарий:
Время отмены превышает допустимый срок: система запрещает отмену.

## 9. Уведомления о бронированиях
### Предусловия:
1. Пользователь авторизован.
2. В системе есть активные бронирования.

### Триггер:
Приближается время начала бронирования.

### Основной сценарий:

1. Система автоматически отправляет уведомление пользователю.
2. Уведомление содержит информацию о ресурсе и времени.

### Результат:
Пользователь своевременно проинформирован о бронировании.

### Альтернативный сценарий:
Уведомления отключены в настройках пользователя.

## 10. Управление ресурсами (администратор)
### Предусловия:
Пользователь авторизован с ролью администратора.

### Триггер:
Администратор открывает раздел «Управление ресурсами».

### Основной сценарий:
1. Система отображает список всех ресурсов.
2. Администратор добавляет, редактирует или удаляет ресурс.
3. Система сохраняет изменения.

### Результат:
Информация о ресурсах обновлена в системе.

### Альтернативный сценарий:
Недостаточно прав доступа: система запрещает выполнение действия.
