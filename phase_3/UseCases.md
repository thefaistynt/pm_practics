Описание проекта
По итогам Фазы 1 был выбран проект «Система бронирования аудиторий, переговорных и техники».
Use Cases
1. Регистрация нового пользователя

Предусловия:

Пользователь впервые заходит в систему.

Триггер:

Пользователь нажимает кнопку «Зарегистрироваться».

Основной сценарий:

Система предлагает ввести email (или корпоративный логин) и пароль.

Пользователь вводит данные.

Система проверяет, что пользователь с таким email не зарегистрирован.

Система отправляет код подтверждения на email.

Пользователь вводит код подтверждения.

Система предлагает выбрать роль (сотрудник / администратор).

Пользователь подтверждает регистрацию.

Система перенаправляет пользователя на главный экран.

Результат:

Создан новый аккаунт пользователя.

Альтернативный сценарий:

Email уже зарегистрирован: система предлагает выполнить вход.

Неверный код подтверждения: система выводит сообщение об ошибке.

2. Авторизация пользователя

Предусловия:

Пользователь зарегистрирован в системе.

Триггер:

Пользователь нажимает кнопку «Войти».

Основной сценарий:

Система запрашивает email и пароль.

Пользователь вводит учетные данные.

Система проверяет корректность данных.

Система авторизует пользователя и открывает главный экран.

Результат:

Пользователь вошёл в систему.

Альтернативный сценарий:

Неверный логин или пароль: система выводит сообщение об ошибке.

3. Просмотр списка доступных ресурсов

Предусловия:

Пользователь авторизован.

Триггер:

Пользователь открывает раздел «Ресурсы».

Основной сценарий:

Система отображает список ресурсов (аудитории, переговорные, техника).

Для каждого ресурса показывается статус доступности.

Пользователь применяет фильтры (тип ресурса, вместимость, дата).

Результат:

Пользователь видит актуальный список доступных ресурсов.

Альтернативный сценарий:

Нет доступных ресурсов: система отображает соответствующее сообщение.

4. Поиск ресурса по параметрам

Предусловия:

Пользователь авторизован.

Триггер:

Пользователь вводит параметры поиска (дата, время, тип ресурса).

Основной сценарий:

Система принимает параметры поиска.

Система анализирует расписание бронирований.

Система выводит список доступных ресурсов.

Результат:

Пользователь получает список ресурсов, подходящих под заданные условия.

Альтернативный сценарий:

Подходящих ресурсов нет: система предлагает изменить параметры поиска.

5. Бронирование аудитории или переговорной

Предусловия:

Пользователь авторизован.

Ресурс доступен на выбранное время.

Триггер:

Пользователь нажимает кнопку «Забронировать».

Основной сценарий:

Система открывает форму бронирования.

Пользователь указывает дату, время начала и окончания.

Пользователь подтверждает бронирование.

Система проверяет отсутствие конфликтов.

Система сохраняет бронирование.

Результат:

Ресурс успешно забронирован.

Альтернативный сценарий:

В выбранное время ресурс уже занят: система предлагает выбрать другое время.

6. Бронирование техники

Предусловия:

Пользователь авторизован.

Техника доступна.

Триггер:

Пользователь выбирает технику и нажимает «Забронировать».

Основной сценарий:

Система показывает форму с датой и временем использования.

Пользователь указывает период бронирования.

Система проверяет доступность техники.

Система подтверждает бронирование.

Результат:

Техника забронирована на выбранный период.

Альтернативный сценарий:

Техника недоступна: система предлагает альтернативные варианты.

7. Просмотр своих бронирований

Предусловия:

Пользователь авторизован.

Триггер:

Пользователь открывает раздел «Мои бронирования».

Основной сценарий:

Система отображает список всех активных и прошедших бронирований.

Пользователь может открыть детали бронирования.

Результат:

Пользователь видит все свои бронирования.

Альтернативный сценарий:

У пользователя нет бронирований: система показывает пустой список.

8. Отмена бронирования

Предусловия:

Пользователь авторизован.

Бронирование активно.

Триггер:

Пользователь нажимает кнопку «Отменить бронирование».

Основной сценарий:

Система запрашивает подтверждение отмены.

Пользователь подтверждает действие.

Система удаляет бронирование.

Ресурс становится доступным для других пользователей.

Результат:

Бронирование отменено.

Альтернативный сценарий:

Время отмены превышает допустимый срок: система запрещает отмену.

9. Уведомления о бронированиях

Предусловия:

Пользователь авторизован.

В системе есть активные бронирования.

Триггер:

Приближается время начала бронирования.

Основной сценарий:

Система автоматически отправляет уведомление пользователю.

Уведомление содержит информацию о ресурсе и времени.

Результат:

Пользователь своевременно проинформирован о бронировании.

Альтернативный сценарий:

Уведомления отключены в настройках пользователя.

10. Управление ресурсами (администратор)

Предусловия:

Пользователь авторизован с ролью администратора.

Триггер:

Администратор открывает раздел «Управление ресурсами».

Основной сценарий:

Система отображает список всех ресурсов.

Администратор добавляет, редактирует или удаляет ресурс.

Система сохраняет изменения.

Результат:

Информация о ресурсах обновлена в системе.

Альтернативный сценарий:

Недостаточно прав доступа: система запрещает выполнение действия.
